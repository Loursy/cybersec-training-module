Adım 3: Teknik Analiz - Sistem Nasıl Çöktü?

4 haneli bir şifrenin güvenli olmasının tek sebebi, insanın 10.000 ihtimali elle deneyemeyecek olmasıdır. Ancak karşımızda bir insan değil, bilgisayar programı vardı. İşte arka planda yaşananlar:

1
Sınırların Olmaması (Lack of Rate Limiting)

Yazılımcı, kullanıcının girdiği kodun doğru olup olmadığını kontrol eden bir API yazdı ama bir kullanıcının o API'ye 1 saniyede kaç kere istek atabileceğini kontrol etmedi.

// ZAFİYETLİ KOD MANTIĞI
if (girilen_kod == gercek_sms_kodu) { giris_yap(); }
else { hata_ver(); } // Sınırsız denemeye açık!
2
Kaba Kuvvetin Gücü (Brute-Force)

Otomatik bir script, HTTP POST istekleri kullanarak API'ye saldırdı. 0000'dan başlayarak her milisaniyede yeni bir kod denedi. Sistem yorulmadı, engellemedi ve sadece "Yanlış" yanıtı döndü. Ta ki HTTP 200 OK (Başarılı) kodunu görene kadar.

🛡️ Kesin Çözüm: Rate Limiting ve Account Lockout

Brute-Force ve Credential Stuffing saldırılarını durdurmanın en kesin yolu API seviyesinde kısıtlama yapmaktır.

1. Rate Limiting: Bir IP adresi 1 dakikada sadece 5 istek yapabilir. Fazlasında sistem HTTP 429 (Too Many Requests) hatası vermelidir.
2. Account Lockout: Arka arkaya 3 hatalı SMS kodundan sonra giriş işlemi 15 dakikalığına tamamen dondurulmalıdır.
