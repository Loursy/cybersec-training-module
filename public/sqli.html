Adım 2: Zafiyet Simülasyonu (Hack Time)

Bu login paneline kullanıcı adı ve şifre gibi istenen verileri girmek yerine bir SQL sorgusu yazıyoruz. Yani bize bırakılan bu boşluktan faydalanarak istediğimiz sorguyu yazıp veritabanını manipüle ediyor ve normalde gizlenmesi gereken bilgilere ulaşıyoruz. :)

Aşağıda güvenlik açığı barındıran zafiyetli bir sistemin giriş paneli bulunmaktadır. Sizden normal bir "E-posta" bekleniyor. Ancak biz buraya kendi ismimizi yazmak yerine, arka planda çalışan veritabanını kandıracak özel bir metin yazacağız.

Siber güvenlikte buna "Payload" (Saldırı Yükü) denir. Hedefe gönderdiğimiz bu metni, sistemin masum sanıp içeri aldığı ama aslında kapıları içeriden açan dijital bir Truva Atı 🐴 gibi düşünebilirsin!

Hadi o Truva Atını sisteme yollayalım. E-posta kısmına tam olarak şunu yazın: ' OR 1=1 --

Peki bu payload ne işe yarayacak? (' OR 1=1 --)

  • ' (Tek Tırnak): Geliştiricinin arka planda yazdığı kodu biz erken kapatıyoruz. Böylece sistemin kodunun içine kendi kelimelerimizi ekleyebiliyoruz.
  • OR 1=1: "Veya 1 eşittir 1" diyoruz. 1 her zaman 1'e eşit olduğu için, veritabanı "Tamam bu kişi doğru söylüyor" diyerek bizi zorla içeri alıyor.
  • -- (İki Tire): Veritabanına "Bundan sonra yazan şifre kontrol kısmını tamamen görmezden gel, sil gitsin" emrini veriyor.

Adım 3: Zafiyet Analizi ve Çözümü

SQL Injection (SQLi) Nedir?

Kötü niyetli kullanıcıların, uygulamanın veritabanına gönderdiği sorgulara müdahale etmesini sağlayan kritik bir güvenlik açığıdır. Geliştirici, kullanıcıdan aldığı veriyi güvenlik süzgecinden geçirmeden doğrudan SQL komutunun içine koyarsa, saldırgan kendi zararlı komutlarını sistemin komutuymuş gibi veritabanına çalıştırttırabilir.

Nasıl Çözülür? (Prepared Statements)

Güvenlik açığının temel sebebi, kullanıcının girdiği metnin bir 'veri' olarak değil, çalıştırılabilir bir 'SQL komutu' olarak algılanmasıdır. Bunu kesin olarak çözmek için Parametreli Sorgular (Prepared Statements) kullanmalıyız.

Hatalı (Zafiyetli) Kod Kullanımı: 

// KÖTÜ KULLANIM: Kullanıcı verisi kodun içine doğrudan gömülüyor
const query = `SELECT * FROM users 
               WHERE email = '${email}' 
               AND password = '${password}'`;

// Eğer kullanıcı payload olarak: ' OR 1=1 -- girerse, 
// Sorgu dinamik olarak çalıştırılabilir bir sızıntı komutuna dönüşür:
db.all(query, (err, rows) => { ... });

Aşağıdaki güvenli kodda görebileceğiniz gibi, dışarıdan gelen verileri doğrudan SQL metnine yazmak yerine ? (soru işareti) koyuyoruz. Veritabanı sürücüsü, bu soru işaretlerinin yerine gelecek verileri otomatik olarak temizler ve sadece düz metin olarak işler.

Güvenli (Doğru) Kod Kullanımı: 

// GÜVENLİ: Doğrudan koda yapıştırmak yerine parametre (?) kullanımı
const query = `SELECT * FROM users 
               WHERE email = ? 
               AND password = ?`;

// Veriler ayrı bir dizi [email, password] olarak gönderilir.
// Kullanıcı saldırı metni yazsa bile, veritabanı bunu bir komut değil, 
// sadece "saçma bir e-posta adresi" sanıp reddedecektir.
db.get(query, [email, password], (err, row) => { ... });
